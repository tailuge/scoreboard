# Code Review Report

This review covers the Next.js API routes and service layers, focusing on security, performance, and maintainability.

---

### 🔴 Critical Issues

#### 1. Lack of Score Verification in High Score Submission
- **File:** `src/pages/api/hiscore.ts` (Lines 77–80)
- **Problem:** The endpoint trusts the `score` and `state` provided by the client without any server-side validation or cryptographic signing. A user can easily manipulate the `json.score` value in the POST body to submit any high score they desire.
- **Rationale:** High scores are the core of the competitive experience. Allowing unverified submissions trivializes the leaderboard.
- **Suggested Solution:**
  - Implement server-side verification of the game state if possible.
  - Alternatively, use a signed hash (HMAC) generated by the game engine that the server can verify to ensure the score was achieved through legitimate gameplay.
  ```typescript
  // In src/pages/api/hiscore.ts
  if (!verifyScoreSignature(json.score, json.signature)) {
    return new Response("Invalid score signature", { status: 403 })
  }
  ```

#### 2. Insecure Tie-Breaker Logic
- **File:** `src/pages/api/hiscore.ts` (Line 79)
- **Problem:** The score is augmented with a time-based decimal: `const score = json?.score + (Date.now() - base) / base`.
- **Rationale:** This allows users to "improve" their score slightly just by waiting longer to submit, as `Date.now()` increases. A competitive leaderboard should ideally favor the first person to achieve a score, or use a more deterministic tie-breaker.
- **Suggested Solution:** If earlier scores should rank higher, subtract the time offset instead, or better yet, use the timestamp as a secondary sort key in Redis.

### 🟡 Suggestions

#### 1. Inefficient Redis Retrieval and Manual Reversal
- **File:** `src/services/scoretable.ts` (Line 42)
- **Problem:** `(await this.store.zrange(this.dbKey(ruletype), 0, 9)).reverse()` fetches the lowest scores first and reverses them in application memory.
- **Rationale:** `@vercel/kv` supports the `rev: true` option in `zrange`, which allows the database to return the results in the correct order directly.
- **Suggested Solution:**
  ```typescript
  // src/services/scoretable.ts
  async topTen(ruletype: string) {
    const data = await this.store.zrange(this.dbKey(ruletype), 0, 9, { rev: true })
    return data.map((row: ScoreData) => ({ ... }))
  }
  ```

#### 2. Potential Scalability Bottleneck in `TableService`
- **File:** `src/services/TableService.ts` (Lines 44, 60)
- **Problem:** `this.store.hgetall<Record<string, Table>>(KEY)` is used to fetch *all* active tables to perform expiration checks and filtering.
- **Rationale:** As the number of active tables grows, `HGETALL` becomes increasingly expensive in terms of network I/O and CPU.
- **Suggested Solution:** Use a separate Redis Sorted Set for table expiration (where the score is the `lastUsedAt` timestamp). This allows querying only for expired tables using `ZRANGEBYSCORE` and avoids fetching the data for all active tables.

#### 3. Redundant `OPTIONS` Request Handling
- **File:** `src/middleware.ts` vs. various API routes (e.g., `src/pages/api/rank/index.ts`)
- **Problem:** The middleware handles `OPTIONS` requests globally (returning 204), but many individual API handlers also have explicit `OPTIONS` checks (returning 200).
- **Rationale:** Redundant logic leads to inconsistency. In this case, the individual handlers are never reached for `OPTIONS` requests because the middleware intercepts them first, making the code in the handlers dead logic.
- **Suggested Solution:** Remove the `OPTIONS` branches from individual API handlers to keep the codebase DRY and rely on the global middleware.

### ✅ Good Practices

#### 1. Effective Use of Edge Runtime
- **Files:** `src/pages/api/match-results.ts`, `src/pages/api/rank/index.ts`, etc.
- **Comment:** Leveraging `export const config = { runtime: "edge" }` is excellent for reducing latency and improving the performance of globally distributed API requests.

#### 2. Robust Client Metadata Capture
- **File:** `src/pages/api/match-results.ts` (Lines 100–120)
- **Comment:** The use of `x-vercel-ip-*` headers and `userAgent(request)` to capture location and browser metadata provides valuable analytics while maintaining privacy (no IP addresses stored).

#### 3. Proactive Cleanup in `MatchResultService`
- **File:** `src/services/MatchResultService.ts` (Lines 23–40)
- **Comment:** The `evictOldResults` method is a great implementation of a rolling history, ensuring that the Redis store doesn't grow unbounded while also cleaning up associated replay data keys.
