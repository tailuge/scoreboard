# Migration Plan: Replace Backend with @tailuge/messaging

## Overview

This document outlines the migration from the current KV-based table system to the `@tailuge/messaging` library for real-time presence, matchmaking, and game communication.

## Current System Summary

The current system uses:

- **KV storage** (`tables` hash) for persistent table state
- **TableService** for matchmaking (find-or-create, join, spectate)
- **NchanPub/NchanSub** for raw WebSocket presence handling
- **LobbyContext** + custom hooks for React integration
- **~10 API routes** for table CRUD operations
- **User tracking**: Single-player tables expire after 1 minute, multiplayer after 10 minutes
- **Table ID**: Serves as join code - players share `/game?tableId=XXX`

## What the Package Provides

- `MessagingClient` - unified entry point with `start()`/`stop()`
- `Lobby` - presence + matchmaking via `challenge()`/`acceptChallenge()`/`onUsersChange()`
- `Table` - game messaging via `publish()`/`onMessage()`
- Helpers: `activeGames()`, `canChallenge()`, `canSpectate()`

## Key Differences & Risk Areas

### 1. Table Lifecycle & Expiration

| Current System                                                          | New System                                                                           |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| Tables stored in KV with explicit expiration (1min single, 10min multi) | Tables exist only as presence data - user with `tableId` = active table              |
| Server-side cleanup via TTL                                             | No explicit table expiration - derived from presence timeout (90s TTL on heartbeats) |
| Tables can be queried independently                                     | Tables only exist when players are present                                           |

**Risk - HIGH**: The new system doesn't have persistent table state. If both players leave, the "table" disappears entirely. This breaks:

- **Match history** - currently stored in KV, needs to remain for historical data
- **Replay functionality** - currently stored in KV, needs to remain for completed games
- **Spectator links to past games** - URL `/game?tableId=XXX` won't work after players leave
- **Single-player practice mode** - may need KV storage for this use case

**Mitigation**: Keep KV storage for match results/replays. The active game list comes from presence, completed games come from KV.

### 2. Join Code / Table Sharing

| Current System                | New System                                                   |
| ----------------------------- | ------------------------------------------------------------ |
| Table ID is the join code     | Challenge creates a table, but sharing mechanism may differ  |
| Share via `/game?tableId=XXX` | Likely use same URL pattern but tableId generated by library |

**Risk - MEDIUM**: The library generates table IDs internally. Need to verify:

- Format compatibility with existing game page routing
- Whether table IDs are deterministic or random
- How to generate shareable links for auto-match (vs direct challenge)

### 3. Matchmaking Behavior (BIGGEST CHANGE)

| Current System                                                   | New System                                     |
| ---------------------------------------------------------------- | ---------------------------------------------- |
| `find-or-create`: Finds any pending table with matching ruleType | `challenge`: Direct challenge to specific user |
| Auto-match to random pending table                               | Must select specific opponent                  |
| No accept/decline flow                                           | Explicit accept/decline/cancel flow            |

**Risk - HIGH**: Auto-match is a core feature. The library's challenge system is peer-to-peer (direct), NOT auto-match. Users can't click "Find Game" and get matched to random opponent.

**Mitigation Options**:

- **Option A**: Implement auto-match by polling for users without `tableId` and with matching `ruleType`, then challenge first available
- **Option B**: Keep a simple KV index for "seeking" players, use Nchan to broadcast seeking status
- **Option C**: Add "seek" flow to library (it has `seek` field in PresenceMessage)

### 4. Spectator Model

| Current System                                          | New System                                                                      |
| ------------------------------------------------------- | ------------------------------------------------------------------------------- |
| Dedicated API to spectate (`/api/tables/[id]/spectate`) | Spectator list derived from presence - any user with `tableId` can be spectated |
| Spectators don't affect table state                     | Library treats all presence with `tableId` as potential spectatable             |

**Risk - LOW**: Need to verify spectator flow. The library may not distinguish between players and spectators in the same way.

### 5. Data Stored per User

| Current System                                                                                            | New System                                       |
| --------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| userId, userName, locale, originUrl, isBot, ua, opponentId, ruletype, timestamp                           | Same + `tableId`, `seek`                         |
| Additional table data: creator, players, spectators, createdAt, lastUsedAt, isActive, ruleType, completed | Table state derived from presence with `tableId` |

**Risk - MEDIUM**: Some current table metadata (createdAt, lastUsedAt, completed, spectators array) is no longer available. May need:

- KV storage for completed game state
- Separate tracking for spectator counts

### 6. Heartbeat & Timeout

| Current System            | New System                          |
| ------------------------- | ----------------------------------- |
| 60s heartbeat             | Library handles this (configurable) |
| 90s TTL for stale removal | Library handles this                |

**Risk - LOW**: These match current defaults, should be compatible.

### 7. Game Page Integration

The library's `Table` class provides `publish()` and `onMessage()` for game events. Current system uses:

| Current System             | New System                             |
| -------------------------- | -------------------------------------- |
| KV storage for game state  | Table messages for real-time sync      |
| Polling for opponent moves | `Table.onMessage()` callback           |
| No explicit game events    | Explicit `publish("MOVE", data)` calls |

**Risk - MEDIUM**: The game page (`/game`) needs refactoring to use `Table.onMessage()` instead of polling. This is future work beyond Phase 1.

### 8. API Routes

All these current routes become unnecessary:

| Route                           | Reason                                      |
| ------------------------------- | ------------------------------------------- |
| POST /api/tables                | Replaced by Lobby.challenge()               |
| GET /api/tables                 | Replaced by activeGames(users)              |
| POST /api/tables/find-or-create | Replaced by Lobby.challenge() / auto-match  |
| PUT /api/tables/[id]/join       | Replaced by Lobby.acceptChallenge()         |
| PUT /api/tables/[id]/spectate   | Replaced by client.joinTable()              |
| PUT /api/tables/[id]/complete   | Still needed for match results storage (KV) |
| DELETE /api/tables/[id]         | Replaced by Lobby.cancelChallenge()         |

**Risk - LOW**: Most API routes can be deleted. Only `/api/match-results` and `/api/tables/[id]/complete` may need to stay for match recording.

## Testing Strategy

- **Unit tests**: Mock `MessagingClient` (Jest mocks)
- **No Nchan required** for unit tests - test React integration with mocked service
- Keep existing test structure in `src/tests/`

## Implementation Order

2. Create new React context using MessagingClient
3. Update Lobby page to use new context
4. Update tests with mocks
5. Delete old files and API routes
6. Run tests and fix any issues
7. Run lint/prettify
